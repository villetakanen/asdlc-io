## P0 — Spec Violations (Blocking)

### 1. `ai-software-factory.md` — Archetype Misclassification

**Spec:** `concept-spec.md` — *"If it's explaining 'how to do X in ASDLC', it's a Pattern or Practice, not a Concept."*

The article's body is structured as Problem → Solution → Mechanics (Non-Interactive Development, Probabilistic Satisfaction, Holdout Scenarios, Digital Twin Universes). This is Pattern anatomy, not Concept terminology definition. The Decision Tree in concept-spec asks: *"Does this term have industry-standard meaning?"* — "AI Software Factory" is largely ASDLC-coined, which the spec says means *"it's probably a Pattern, not a Concept."*

**Recommendation:** Either (a) restructure as a Pattern article with proper Problem/Solution/Anatomy sections per `pattern-spec.md`, or (b) radically trim to a Concept that defines "Software Factory" as an industry term (from manufacturing/DevOps lineage) with a brief ASDLC Usage section linking to a new pattern.

### 2. `digital-twins.md` — Archetype Misclassification

**Spec:** Same as above.

The article uses Pattern structure: "The Problem" → "The Agentic Solution". Concept Archetype A requires: Definition → Key Characteristics → ASDLC Usage. Additionally, "Digital Twin" is a well-established industry term (manufacturing, IoT, simulation) that the article completely ignores — jumping straight to the ASDLC-specific meaning.

**Spec:** *"Start with industry definition, then explain ASDLC usage."*

**Recommendation:** Rewrite as Archetype A concept: define "Digital Twin" from its manufacturing/simulation origins, explain key characteristics, then add ASDLC Usage section showing how the factory model applies the concept. Alternatively, reclassify as a Pattern.

### 3. Both New Concepts — Missing Required "ASDLC Usage" Section

**Spec:** `concept-spec.md` §5 — *"ASDLC Usage (Required for Archetype A)"*

Neither `ai-software-factory.md` nor `digital-twins.md` contains an explicit "ASDLC Usage" section. The content is woven into the body, but the spec requires a discrete section with grounding links.

### 4. `agent-optimization-loop.md` — Inline External Reference

The new subsection links directly to `https://factory.strongdm.ai/techniques/probabilistic-satisfaction` in the body without a corresponding entry in the frontmatter `references` array. All external citations must be in frontmatter.

## P1 — Spec Drift (Should Fix)

### 5. `adversarial-code-review.md` — Practice Content in Pattern Article

**Spec:** `pattern-spec.md` — *"Pattern articles MUST describe structure, not process."*

The new §3b "Moderator Synthesis" is step-by-step procedural content:
> 1. The Moderator gathers all Critic reports.
> 2. It deduplicates overlapping findings across lanes.
> 3. It creates a unified, prioritized checklist...
> 4. Important Phase Separation: ...

This is Practice material. The Pattern should describe the *shape* of the Moderator role (it exists, its purpose, its position in the architecture), then link to a Practice for the process.

### 6. `provenance.md` — Implementation Details in Concept

The Identity Separation addition includes tool-specific configuration (`CLAUDE_GITEA_TOKEN` vs `GEMINI_GITEA_TOKEN`). Concept articles define terminology; implementation-level token naming belongs in a Practice.

**Recommendation:** Keep the principle of Identity Separation. Remove the specific token examples and reference a Practice or Pattern for implementation details.

### 7. `adversarial-code-review.md` — Undefined Terminology

"Tri-Model Lane", "Architect Lane", "SecOps Lane", "QA Lane" are introduced without prior establishment as concepts or defined terms. If these are reusable constructs, they need conceptual grounding. If they're just examples, the language should frame them as illustrative, not canonical.

### 8. `ai-software-factory.md` — Internal Contradiction

Line 71 claims holdout scenarios + digital twins make sustained L4 operations *"mathematically safe."* The immediately following section (Governance & Liability Gaps) documents *"severe, unpriced regulatory and legal risks"* at L4. The word "mathematically" makes an extraordinary formal claim that the article itself refutes.

**Recommendation:** Replace "mathematically safe" with "empirically defensible" or "operationally viable" and qualify with "from a technical verification standpoint."

## P2 — SEO & Language (Consider)

### 9. `ai-software-factory.md` — Title SEO Weakness

"AI Software Factory" is ASDLC-coined jargon. Search volume likely clusters on "software factory" (DevOps/manufacturing term) or "agentic software development." The description (162 chars) is functional but ASDLC-specific rather than a pure definition.

**Recommendation:** Consider whether the canonical term should be "Software Factory" (industry-grounded) with "AI Software Factory" and "Dark Factory" as aliases in the body.

### 10. `digital-twins.md` — SEO Term Hijacking

"Digital Twins" has enormous existing search volume tied to IoT, manufacturing, and simulation (Gartner, Siemens, GE). Someone searching this term expects that context. The article offers zero acknowledgment of the established meaning.

**Risk:** Search engines may penalize or deprioritize content that redefines a well-established term without acknowledging prior art. At minimum, this creates a poor user experience for organic visitors.

**Recommendation:** Open with the industry-standard definition of Digital Twins (virtual replicas of physical systems), then pivot to how ASDLC extends this concept to software service simulation. This aligns with concept-spec requirements *and* captures broader search intent.

### 11. `digital-twins.md` — Description Not a Pure Definition

Current: *"Creating high-fidelity, in-memory clones of third-party services using agents to enable high-volume, automated scenario testing without API rate limits."*

This reads as an action/practice description, not a canonical definition. Per concept-spec: *"description: Pure semantic definition (this appears in search results)."*

**Suggestion:** *"A simulated testing environment using behavioral clones of third-party services, enabling high-volume verification without external API dependencies."*

### 12. `further-reading.astro` — Superlative Language

Descriptions use editorial superlatives: *"devastatingly insightful"*, *"exceptional empirical case study"*, *"profound look."* While the further-reading page has looser constraints than knowledge base articles, this tone conflicts with the authoritative, neutral voice the project establishes.

**Recommendation:** Replace with factual descriptors. "Devastatingly insightful legal critique" → "Legal and regulatory critique." "Exceptional empirical case study" → "Empirical case study." Let the content speak for itself.

### 13. `levels-of-autonomy.md` — Legal Jargon Density

The expanded L4 warning adds substantial legal language (product liability, licensing, contractual warranties, accountability mechanisms). While valuable, this dilutes the article's primary SEO purpose as a definitional anchor for "Levels of Autonomy." Consider whether this governance material should live in the ai-software-factory article instead, with a brief cross-reference here.

## Verdict
