# Spec: Content Articles

## Context

ASDLC publishes three content collections — **Concepts**, **Patterns**, and **Practices** — each serving a distinct knowledge function. Together they form a layered knowledge base: concepts define terminology, patterns define structure, practices define process.

The three collections share a common technical foundation (Zod schema, Astro content layer, article layouts, MCP exposure) but differ in purpose, tone, and content structure. This spec defines the shared contract; per-collection archetypes are documented separately:

- [Concept Archetype](concept.md) — Terminology authority
- [Pattern Archetype](pattern.md) — Architectural blueprints
- [Practice Archetype](practice.md) — Operational processes

## Architecture

- **Content Source** — Markdown/MDX files in `src/content/{concepts,patterns,practices}/`.
- **Schema** — `articleSchema` base in `src/content/config.ts`, extended per collection where needed (e.g., `supersededBy` on concepts).
- **Routing** — File-based: `src/pages/{concepts,patterns,practices}/[...slug].astro` using `getStaticPaths()`.
- **Layout** — `ArticleLayout.astro` renders frontmatter-driven header (`SpecHeader`), body content, structured references (`ArticleReferences`), and Schema.org metadata.
- **Index Pages** — Per-collection index at `/{collection}/` using `SpecCard` and/or `SpecListItem`.
- **Field Manual** — Alphabetic compilation of Live/Experimental articles across all collections.
- **MCP** — All Live/Experimental articles exposed via `list_articles`, `get_article`, `search_knowledge_base`.
- **Downloadable Skill** — Static markdown bundle generated by `pnpm build:skill`.

### Collection Classification

When creating content, use this decision tree to select the correct collection:

1. **Does this define a term or clarify semantics?** → Concept
2. **Does this describe a reusable structure or architectural form?** → Pattern
3. **Does this provide step-by-step operational instructions?** → Practice

Cross-check:
- "What does X mean?" → Concept
- "What is the shape of the solution?" → Pattern
- "How do I do X?" → Practice

If content spans two categories, split it. A pattern that includes step-by-step instructions should have its process extracted to a linked practice.

### Shared Frontmatter Contract

All content articles share these fields (from `articleSchema`):

```yaml
---
title: "Article Title"           # Collection-appropriate casing (see archetypes)
description: "Summary..."        # Max 200 chars, clear standalone definition
tags: ["Tag1", "Tag2"]           # 2-5 industry keywords
relatedIds: ["collection/slug"]  # Bidirectional cross-references
lastUpdated: 2025-01-15          # ISO 8601 date
status: "Live"                   # Live | Experimental | Draft | Proposed | Deprecated
references: []                   # External sources (see article-references spec)
---
```

### Cross-Referencing Rules

- `relatedIds` links are **bidirectional** — if A references B, B should reference A.
- Internal links use collection-prefixed slugs: `concepts/context-engineering`, `patterns/the-spec`.
- External sources go in `references` frontmatter, never in `relatedIds`.
- A resource appears in either `relatedIds` or `references`, never both.

## Anti-Patterns

- **Collection Misclassification** — Putting "how-to" content in Patterns, or structural definitions in Practices. Use the decision tree.
- **One-Way Links** — Adding a `relatedId` without updating the target article's frontmatter.
- **Body References Section** — Using a markdown `## References` heading instead of the `references` frontmatter array.
- **H1 in Content** — Using `#` headings in article body. H1 is reserved for the layout; content starts at `##`.
- **Marketing Language** — "Revolutionary", "game-changing", "next-generation". State facts.
- **Schema Drift** — Hardcoding frontmatter fields not defined in `articleSchema`. All fields must validate against `src/content/config.ts`.
- **Orphaned Content** — Articles with zero `relatedIds` and zero `references`. Every article must connect to the knowledge graph.

## Contract

### Definition of Done

- The article validates against `articleSchema` in `src/content/config.ts` (`pnpm check` passes).
- The article builds without errors (`pnpm build` succeeds).
- The article lint passes (`pnpm lint` succeeds).
- Frontmatter `description` is under 200 characters and is a clear standalone definition.
- Tags are 2–5 industry keywords (not ASDLC-internal jargon).
- All `relatedIds` are bidirectional — the referenced article links back.
- No `#` (H1) headings appear in the markdown body.
- Mermaid diagrams, if used, have been rendered via `pnpm diagrams`.
- The article appears on its collection index page and in the Field Manual (if Live/Experimental).
- The article is retrievable via MCP `get_article` (if Live/Experimental).
- The article conforms to its collection archetype (see per-type specs).

### Regression Guardrails

- All content must validate against `articleSchema` — no unschematized frontmatter fields.
- The `description` field must remain capped at 200 characters.
- Dates must use `z.coerce.date()` — no string dates.
- The `references` field must remain optional with empty-array default.
- Deprecated articles must specify `supersededBy` in frontmatter.
- External links in article body are permitted only for inline context; attributable sources must use `references`.
- Content must start at H2 (`##`) — the H1 is rendered by the layout from `title`.

### Scenarios

**Scenario: New article creation**
- Given a topic that fits one of the three collections
- When an author creates the article following the appropriate archetype
- Then the article validates, builds, and appears on the correct index page

**Scenario: Cross-reference integrity**
- Given article A adds `relatedIds: ["patterns/the-spec"]`
- When the change is reviewed
- Then `patterns/the-spec` also lists article A in its `relatedIds`

**Scenario: Collection boundary enforcement**
- Given an article contains both structural definitions and step-by-step instructions
- When reviewed for compliance
- Then the structural content is extracted to a Pattern and the instructions to a Practice, linked via `relatedIds`

**Scenario: Deprecated article handling**
- Given an article's status changes to Deprecated
- When the frontmatter is updated
- Then `supersededBy` references the replacement article(s)
- And the article is excluded from the Field Manual

**Scenario: MCP discoverability**
- Given a Live article exists in any collection
- When an LLM calls `search_knowledge_base` with a relevant query
- Then the article appears in the search results
