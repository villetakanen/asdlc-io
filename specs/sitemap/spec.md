# Feature: Sitemap lastmod from content freshness

## Blueprint

### Context

The site's XML sitemap is generated by `@astrojs/sitemap` (configured in `astro.config.mjs`) and served at `/sitemap-index.xml` → `/sitemap-0.xml`. As of this spec, the sitemap emits `<loc>` entries but **no `<lastmod>`**.

Every article in the content collections already carries a Zod-validated `lastUpdated` date (`src/content/config.ts`, `articleSchema.lastUpdated`). The freshness data exists — it is simply not plumbed through to the sitemap.

`<lastmod>` is a primary crawl-prioritization signal. Search and AI crawlers (Googlebot, GPTBot, ClaudeBot, PerplexityBot) use it to decide which URLs to re-fetch and how often. Emitting an accurate `lastmod` per article lets crawlers re-crawl updated articles promptly and skip unchanged ones, improving citation freshness with zero ongoing authoring cost.

This is item **H3** in `docs/reports/GEO-AUDIT-REPORT.md` and a sub-task of the GEO Optimization Pass epic (AL-55).

### Architecture

**Single touch point:** `astro.config.mjs` — the `sitemap()` integration call gains a `serialize` callback.

**The mapping problem.** `astro.config.mjs` runs in the Vite/Node config context, where `astro:content`'s `getCollection()` is **not** available. The `lastUpdated` values must therefore be read directly from the markdown frontmatter at config-load time, building a lookup keyed by URL path.

**Data flow:**

```
src/content/{concepts,patterns,practices,recipes}/**/*.md
        │  (read at config load via fs glob + gray-matter)
        ▼
  Map<urlPath, lastmodISO>     e.g. "/concepts/levels-of-autonomy/" → "2026-04-10T00:00:00.000Z"
        │
        ▼
  sitemap({ serialize })  — for each item, parse pathname, look up the map,
                            set item.lastmod when a match exists; otherwise leave it unset
        ▼
  /sitemap-0.xml  with <lastmod> on every article URL
```

**URL derivation.** All four article collections route as `/{collection}/{id}/`:

| Collection | Route file | URL shape |
|---|---|---|
| concepts | `src/pages/concepts/[...slug].astro` (`slug: entry.id`) | `/concepts/{id}/` |
| patterns | `src/pages/patterns/[...slug].astro` | `/patterns/{id}/` |
| practices | `src/pages/practices/[...slug].astro` | `/practices/{id}/` |
| recipes | `src/pages/recipes/[id].astro` | `/recipes/{id}/` |

The glob-loader `id` is the file path relative to the collection base, minus the `.md` extension (e.g. `foo.md` → `foo`, `nested/bar.md` → `nested/bar`). The lookup must normalize trailing slashes so it matches whatever form `@astrojs/sitemap` emits.

**Date format.** `lastmod` is emitted as a W3C / ISO-8601 datetime string (`Date.prototype.toISOString()`), derived from the frontmatter `lastUpdated`. gray-matter parses unquoted YAML dates (`lastUpdated: 2026-04-10`) into JS `Date` objects, so the value is normalized through `new Date(...)` before serialization.

**Tooling.** `gray-matter` (already present as a dependency, used by `scripts/lint-specs.mjs`) parses frontmatter. Node's built-in `node:fs` + the `glob` package (already a dependency) enumerate the files. **No new dependency is added.**

### Anti-Patterns

- **Do NOT import `astro:content` / `getCollection` in `astro.config.mjs`.** It is unavailable in the config context and will fail the build.
- **Do NOT hardcode a single site-wide `lastmod`** (e.g. build date). The signal must be per-article and reflect actual content freshness, or it is noise.
- **Do NOT fabricate `lastmod` for non-article pages.** Home, collection index pages, and `resources/*` have no `lastUpdated` — leave their `lastmod` unset rather than inventing a date.
- **Do NOT duplicate the URL-routing logic** beyond the `/{collection}/{id}/` rule already encoded in the page routes. If routing changes, this map must change with it (see Regression Guardrails).
- **Do NOT add a second source of truth for freshness.** `lastUpdated` frontmatter remains the only authority; the sitemap reads it, never overrides it.

## Contract

### Definition of Done

- [ ] `astro.config.mjs` passes a `serialize` callback to `sitemap()` that emits `<lastmod>` for article URLs.
- [ ] After `pnpm build`, every article URL (`/concepts/*`, `/patterns/*`, `/practices/*`, `/recipes/*`) in `dist/sitemap-0.xml` has a `<lastmod>` element.
- [ ] Each emitted `<lastmod>` equals the `lastUpdated` from that article's frontmatter (date-equal, ISO-8601 form).
- [ ] `grep -c lastmod dist/sitemap-0.xml` ≥ 70 (73 articles exist at authoring time).
- [ ] Non-article pages (home, index pages, `resources/*`) either carry no `<lastmod>` or are unaffected — the build does not crash on them.
- [ ] No new package is added to `package.json`.
- [ ] `pnpm build` and `pnpm check` pass.

### Regression Guardrails

- The `site: "https://asdlc.io"` config and the `sitemap()` integration MUST remain — `robots.txt` declares `Sitemap: https://asdlc.io/sitemap-index.xml` and depends on it.
- The URL-mapping rule MUST stay in sync with the page routes (`/{collection}/{id}/`). If a collection's route changes, the sitemap map breaks silently (URLs get no `lastmod`) — the `grep -c lastmod ≥ 70` check is the canary.
- `lastUpdated` MUST remain the sole freshness authority; this feature is read-only over content.
- The map build MUST NOT throw on a content file that is missing or has an unparseable `lastUpdated` — degrade to "no lastmod for this URL", never fail the whole build. (A missing `lastUpdated` is already caught by the content-collection Zod schema at build time; the sitemap path must not be the thing that surfaces it.)

### Scenarios

**Scenario: Article gains a lastmod**
- Given: `src/content/concepts/levels-of-autonomy.md` has `lastUpdated: 2026-04-10`
- When: `pnpm build` runs
- Then: `dist/sitemap-0.xml` contains a `<url>` for `https://asdlc.io/concepts/levels-of-autonomy/`
- And: that `<url>` has `<lastmod>2026-04-10T00:00:00.000Z</lastmod>` (date-equal to frontmatter)

**Scenario: Editing an article updates its lastmod**
- Given: an article's `lastUpdated` is bumped to a newer date
- When: the site is rebuilt
- Then: that URL's `<lastmod>` reflects the new date, and no other URL's `<lastmod>` changes

**Scenario: Non-article page has no fabricated freshness**
- Given: the homepage `/` and collection index `/concepts/` have no `lastUpdated` frontmatter
- When: the sitemap is generated
- Then: those URLs are still present in the sitemap
- And: they carry no `<lastmod>` (no invented date)

**Scenario: Crawler re-crawl prioritization**
- Given: a crawler has previously fetched the sitemap
- When: it re-reads `/sitemap-0.xml` and compares `<lastmod>` values to its last crawl
- Then: it can identify which articles changed and prioritize re-fetching only those

### Verification

```bash
pnpm build
grep -c '<lastmod>' dist/sitemap-0.xml          # expect ≥ 70
grep -A1 'concepts/levels-of-autonomy' dist/sitemap-0.xml   # spot-check a known date
```

## Implementation Notes

- **Location:** `astro.config.mjs` only. Extract the map-builder into a small helper (inline function or a sibling `.mjs` module) for readability, but keep it within the config's reach.
- **Reading files:** `glob("src/content/{concepts,patterns,practices,recipes}/**/*.md")` → for each, `matter(readFileSync(...))` → derive `{ collection, id }` from the path → key `/{collection}/{id}/` → value `new Date(data.lastUpdated).toISOString()`.
- **Matching in `serialize`:** the callback receives `{ url, ... }` where `url` is absolute (`https://asdlc.io/concepts/foo/`). Parse `new URL(item.url).pathname`, normalize trailing slash, look up the map; if found, return `{ ...item, lastmod }`, else return `item` unchanged.
- **Date safety:** wrap per-file parsing in try/catch so one bad file can't abort the build (see guardrail).
- Canonical reference for the `lastUpdated` field: `src/content/config.ts`. Canonical reference for routes: `src/pages/{concepts,patterns,practices}/[...slug].astro` and `src/pages/recipes/[id].astro`.
